Design Decisions
Mobile-first
The initial drill (Part 1) must work perfectly on mobile. Most users will arrive from mobile ads.
Message Alignment
If the ad says "GTO Solver" and onboarding says "test your level", there's dissonance. Part 0 aligns the message with the experience.
One Choice Only
Users from ads get the game pre-selected. Only organic/direct traffic users make a game choice. No "Both" option.

Design Decisions
Test Mode with Reveal
Frequencies hidden during decision, revealed after. Creates suspense, honest testing, and learning moments. Showing frequencies would make it training, not testing.
15 Single-Street Spots
~20 sec each, ~5 min total. Enough for tendencies (not precise leaks), short enough to maintain engagement. Exact number is A/B testable (10 vs 15 vs 20).
Honest Results
Poker players think in sample sizes. "3 leaks from 15 spots" kills credibility. Instead: observable tendencies + "This was a sample." Builds trust.
Breakdown by Spot Type
NOT by street. With 15 spots across streets, "River accuracy: 33%" is meaningless. Spot type grouping (3bet, single raised, multiway) is more useful and honest.
Gate with Two Options
User self-selects intent at sign-up: "Import hands" = advanced/motivated, "Keep training" = exploratory. Both sign up, but with different intents for Part 2.
Adaptive Framing
High score: "edges that separate good from elite." Medium: "patterns worth exploring." Low: "highest-impact improvement areas." Never punitive, always aspirational.
2A
Part 2A Real Product (Import Hands)
User Flow
1
Guided Tracker setup: download and install
2
Play 1 short session with Tracker running
3
"Your first real report is ready" - aha moment with the user's actual data
4
Continues in product: Replayer, Reports, Library
Design Decisions
Highest-Value Path
Real hands + real report = strongest "aha moment". Much stronger than any pre-selected drill. Goal: get to real report as fast as possible.
Tracker on Free Plan
Path A only works if Tracker works on free (even limited to 10 hands/day). Decision needed: if not free, Path A must lead to upgrade + tracker.
2B
Part 2B Exploration (More Drills)
User Flow
1
Drills focused on tendencies observed during onboarding
2
Introduces Trainer and Training Modules
3
Each drill gives feedback and refines user's profile
!
After X drills: "Want to see this with your actual hands? Download the Tracker"
Design Decisions
Safety Net
Not everyone is ready to install a tracker on day 1. Path B maintains engagement with drills until they're ready for the real product.
Player Profile
Investment loop (Hooked model): detected level, identified leaks, drill progress, session history. The more they use it, the more valuable the profile. Leaving = losing progress.

Design Decisions
Progressive Disclosure
Start simple, complicate with time. Each feature has a contextual mini-onboarding discovered in context of use, not in a feature tour.
AI-lite First
Phase 1: template-based messages with dynamic variables (name, level, leaks). Perceived personalization nearly identical to AI-full, complexity 10x lower.
Returning Users
"Already have an account? Log in" visible but not dominant. Returning users skip drill, see "Welcome back. Here's what changed."

Design Decisions
Zero-Friction Re-entry
Deep links directly to drills. No login screens, no menu navigation. One tap from email to value.
User's Own Data
Every re-engagement message uses the user's actual metrics and progress. Never generic. This is the core principle across all phases.
Foundation
Cross-cutting Principles
Rules that apply across every phase of the onboarding architecture.
Multi-game: NLHE + PLO. Identical flows, game-specific content
Honesty: 15 spots = tendencies, not diagnosis. Never inflate results
Every interaction uses THE USER'S DATA (never generic)
Always deliver value before asking for something
Each feature discovered in context of use, not in a tour
AI-lite first, AI-full later
Player Profile is the asset users build and don't want to lose
Mobile-first on Part 1 (drill must work on phone)
Framing is always aspirational, never punitive
Test mode with reveal (frequencies hidden during decision)
Single-street spots (not full hands) for speed and variety
Designed for A/B testing at critical decision points
Testing
Priority A/B Tests
Experiments to validate key assumptions and optimize conversion.
1
10 vs 15 vs 20 Spots
Completion rate vs quality of detected tendencies. Finding the sweet spot between engagement and statistical value.
2
Gate: Two Options vs One
Sign-up rate and activation quality. Does offering a choice improve intent-based routing?
3
Result Framing
Different copy by performance level. Testing aspirational vs neutral framing on conversion.
4
Path B: Drills vs Days
Push to tracker after X drills vs Y days. When to convert explorers into real product users.
5
Per-Spot Feedback
With vs without reveal after each spot. Does immediate feedback increase or decrease completion?
6
High Stakes Bypass
Skill check vs explore product for advanced players. Which path converts better at high stakes?

Open Decisions
Items that need team input before implementation can proceed.
1
Does the Tracker work on the free plan?
Impact: Part 2 Path A viability. If no free tracker, Path A must lead to upgrade first.
2
Who curates the pre-selected spots and how often?
Impact: NLHE + PLO = two separate pools. Needs ongoing curation and quality control.
3
Email/notification infrastructure: exists or needs to be built?
Impact: Parts 3 and 4 depend entirely on email/push capability for progressive onboarding and re-engagement.
4
Timeline for AI-full (Part 3 Phase 2)?
Impact: Determines when to invest in LLM infrastructure vs staying with AI-lite templates.
5
Features available for NLHE at launch: same as PLO or subset?
Impact: Defines scope of multi-game launch. Subset = faster launch, same = consistent experience.
6
"Explore the product" flow for High Stakes bypass: what do they see?
Impact: Needs dedicated UX for experienced players. Database quality, solver speed, replayer UI demo.
